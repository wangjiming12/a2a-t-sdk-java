# 1 a2a-t-sdk-java 开发指南

## 1.1 特性介绍

见[用户指南 1.1特性介绍](https://github.com/project-openan/a2a-t-sdk-java/blob/main/docs/zh/%E7%94%A8%E6%88%B7%E6%8C%87%E5%8D%97.md#11-%E7%89%B9%E6%80%A7%E4%BB%8B%E7%BB%8D)章节。

## 1.2 约束与限制

1. JDK 要求为 17+。
2. 当前支持 `classpath` 和 `local_file` 提示词资源，默认从 jar 内置资源加载。
3. 默认支持 `local_rule` 和 `openai` LLM provider。
4. 协商状态存储当前支持 `in_memory`。
5. SDK 不负责启动业务 HTTP 服务、用户认证、密钥管理和注册中心部署。

### 1.3.1 获取源码

```bash
cd a2a-t-sdk-java
```

### 1.3.2 编译工程

```bash
mvn -DskipTests package
```


```bash

```

如需自动格式化：

```bash

```

## 1.4 Maven 引用

业务项目可使用 BOM 管理版本：

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>net.openan.a2a-t.sdk</groupId>
            <artifactId>a2a-t-bom</artifactId>
            <version>1.0.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

客户端 Agent 引用：

```xml
<dependency>
    <groupId>net.openan.a2a-t.sdk</groupId>
    <artifactId>a2a-t-client</artifactId>
</dependency>
```

服务端 Agent 引用：

```xml
<dependency>
    <groupId>net.openan.a2a-t.sdk</groupId>
    <artifactId>a2a-t-server</artifactId>
</dependency>
```

如果业务系统需要直接读取 SDK 内置资源，可额外引用：

```xml
<dependency>
    <groupId>net.openan.a2a-t.sdk</groupId>
    <artifactId>a2a-t-resources</artifactId>
</dependency>
```

## 1.5 配置加载

Java SDK 不自动发现 `.env`，调用方必须显式传入路径：

```java
import java.nio.file.Path;
import net.openan.a2at.sdk.client.A2ATClient;

A2ATClient client = new A2ATClient(Path.of("client.env"));
```

基础配置示例：

```properties
A2AT_LANGUAGE=zh-CN
A2AT_PROMPT_SOURCE_TYPE=classpath
A2AT_PROMPT_RESOURCE_LOCAL_ROOT_DIR=
A2AT_LLM_PROVIDER=openai
A2AT_LLM_MODEL=deepseek-chat
A2AT_LLM_BASE_URL=https://api.deepseek.com
A2AT_LLM_API_KEY={your_api_key}
A2AT_NEGOTIATION_STATE_STORE_TYPE=in_memory
```


## 1.6 SDK 基本用法

### 1.6.1 客户端生成任务提示词

```java
import java.nio.file.Path;
import net.openan.a2at.sdk.client.A2ATClient;
import net.openan.a2at.sdk.client.model.PromptGenerationResult;

A2ATClient client = new A2ATClient(Path.of("client.env"));
PromptGenerationResult result = client.generateTaskPrompt(
        "通知主题为Incident的，订阅条件为：订阅级别为critical的ETH-LOS的故障，上报通知数据格式为：DataPart");

if (result.success()) {
    System.out.println(result.promptText());
} else {
    System.out.println(result.failure().message());
}
```

### 1.6.2 服务端校验任务提示词

```java
import java.nio.file.Path;
import net.openan.a2at.sdk.server.A2ATServer;
import net.openan.a2at.sdk.server.model.PromptComplianceResult;

A2ATServer server = new A2ATServer(Path.of("server.env"));
PromptComplianceResult result = server.checkTaskPrompt(processedPromptText);

if (result.success()) {
    System.out.println("prompt check passed");
} else {
    System.out.println(result.failure().message());
}
```

### 1.6.3 协商接口

```java
import java.nio.file.Path;
import java.util.Map;
import net.openan.a2at.sdk.client.A2ATClient;
import net.openan.a2at.sdk.negotiation.types.model.NegotiationType;

A2ATClient client = new A2ATClient(Path.of("client.env"));
Map<String, Object> payload = client.startNegotiation(
        NegotiationType.INFORMATION,
        "Please provide incident level.",
        Map.of("missingFields", java.util.List.of("subscription_condition_incident_level")));
```

业务系统需要把协商 payload 中的文本和上下文随 A2A 消息传递给对端，后续通过 `receiveNegotiation` 和 `continueNegotiation` 推进状态。


## 1.7 完整集成开发流程

### 1.7.1 客户端 Agent

客户端 Agent 推荐流程：

1. 读取 `.env` 并创建 `A2ATClient`。
2. 根据用户输入调用 `generateTaskPrompt`。
3. 将 prompt 放入 A2A 消息正文或扩展字段。
4. 通过注册中心查询目标 AgentCard。
5. 根据 AgentCard 中的接口地址发起 A2A 请求。
6. 如果收到协商上下文，通过 `receiveNegotiation` 和 `continueNegotiation` 继续交互。

`a2a-t-sample` 中 `ClientSampleFlow` 展示了完整流程：读取场景、查询 AgentCard、生成 prompt、构造 A2A 请求并处理流式事件。

### 1.7.2 服务端 Agent

服务端 Agent 推荐流程：

1. 创建业务 HTTP 服务或接入已有 A2A Java 服务框架。
2. 启动时构造 AgentCard 并注册到注册中心。
3. 收到 A2A 请求后提取 processed task prompt。
4. 调用 `A2ATServer.checkTaskPrompt`。
5. 校验通过后执行业务逻辑。
6. 校验失败或信息不足时，通过协商接口返回补充信息要求。

`a2a-t-sample` 中 `ServerSampleMain` 和 `ServerSampleFlow` 展示了 AgentCard 注册、HTTP 服务启动、prompt 提取、校验和任务事件推送。

### 1.8.2 如何定义Prompt模板
#### 1.8.2.1 核心价值
A2A-T 结构化Prompt提供了一种可重复使用的结构化方法，用于为LLM提供明确且一致的提示。通过将核心逻辑与可变数据分离，它使智能体间的交互更加可靠/高效/可扩展。使用结构化Prompt的主要好处包括：
- 	一致性：确保提示遵循标准化格式，从而使得智能体的输出更加可预期。
-	效率：避免从头开始编写每个Prompt，节省了时间和精力。同时，也避免了重复复杂的指令。
-	可扩展性：使得为各种业务场景生成prompt更容易。
-	优化：允许对模板进行改进和优化，从而获得更好的结果。

#### 1.8.2.2 A2A-T Prompt模板的分类
面向电信领域的智能体通信，为保证请求内容的完备性，提高推理效率和准确性，A2A-T面向各个AN高价值场景，定义了结构化Prompt模板，并在TMF发布了产业的A2A-T协议标准：
《IG1453A_Structured_Prompt_of_Agent_to_Agent_Protocol_for_Telecoms_A2AT_v1.0.0》
《IG1453_Agent_to_Agent_Protocol_for_Telecoms_A2AT_v2.0.0》

结构化Prompt模板定义分为两层：
- L0基础模板：
定义ICT领域任务的结构化Prompt的基础框架，不针对特定场景的变量和本体进行具体说明。
L0模板列表：

	| 模板名称 | 说明   |
	|--|--|
	|Task-T  | 定义了ICT领域任务的基本结构，但未针对特定场景规定常用变量和通用本体规范。基础模板的解析依赖LLM的推理能力和Agent的上下文处理能力。 |
	|Notification-T |定义了ICT领域一种结构化的基于提示词的网络事件订阅与报告机制。该机制确保对网络事件进行实时感知，并通过结构化提示，在不同层级、不同领域之间提供一致的任务描述。 |

- L1价值场景Prompt模板：
在L0模板的基础上，为不同高价值场景任务定义了常用的“变量”，以便在任务生成过程中，让智能体能够根据这些变量输入相应的内容，并在任务执行过程中识别相关内容以提高推理效率和准确性。

#### 1.8.2.3 核心组成元素
完整的A2A-T Prompt模板一般包含以下两部分：
1. 指令
	- 定义：核心指示或上下文。
	- 作用：提供任务的基本要求和框架。
	- 语法：使用##直接标记指令名称（如## 任务描述）。
2. 变量
	- 定义：动态槽位，每次使用时填充特定数据。
	- 作用：提供更具体的信息，提升推理效率。
	- 语法：使用双大括号{{}}标识变量名（如{{故障发生时间}}）。

##### 1.8.2.3.1 指令

1. 指令语法要求：在声明“指令”时，应使用“##”进行标记，其后是“指令”的名称，以便Agent能够识别，从而实现内容的输入或相应的推理和执行。
2. 指令集合：A2A-T定义的结构化Prompt已为ICT任务Prompt模板奠定了基础框架，将ICT任务的典型信息解构为以下指令。


|指令名称 | 必选/可选|描述与示例 |
|--|--|--|
| 任务描述 |	必选 	| 描述任务的基本要求。例：<br> `## 任务描述` <br> `请根据“目标对象”、“任务上下文”和“约束条件”分析故障的根本原因，并提供修复建议。请按照“预期输出”中定义的结构回应任务。`
|任务类型|可选  |标识任务类型（如故障诊断、能效优化）。例：<br>`## 任务类型`<br>`故障诊断 ` |
|目标对象|可选|  描述任务操作的直接对象。例：<br> `## 目标对象`<br>`故障标识符（fault-csn）为“OSS-FAULT-20250405-001”。`|
|任务上下文|可选  |提供任务执行的背景信息。例：<br>`## 任务上下文`<br>`故障发生时间（occur-time）为“2025-04-05T14:30:00Z”。 ` |
| 预期输出|可选  | 定义预期结果的格式。例：<br>`## 预期输出`<br> `故障诊断结果应包括以下信息：1. 诊断状态：成功或失败 2. 故障诊断分析结果 3. 修复建议 4. 故障根本原因列表 5. 领域特定信息` |

##### 1.8.2.3.2 变量
1. 变量语法要求：在使用“变量”时，应使用双大括号“{{}}”进行标识，并将变量名置于双大括号内，这样智能体才能识别它，从而实现内容的输入或相应的推理和执行。
2. 变量的实例化方法：变量需要正确实例化才能发挥价值，推荐两种方法
	- 自然语言主谓宾结构：故障发生于2026年1月8日16:38:18
	- key-value简洁格式：故障发生时间：2026/1/8/16:38:18
3. 常用变量集合：
A2A-T基于最佳实践，总结了AN L4高价值场景中常用的变量，这些变量有助于有效描述AN L4高价值场景中的任务：
	|变量名称 | 必选/可选|描述与示例 |
	|--|--|--|
	| 标识符 |	必选 	| 用于指定与任务关联的目标标识符。<br> 例如：<br>`## 目标对象`<br> `{{标识符}}` <br>其实例示例如下： <br>`## 目标对象`<br>`故障标识符（fault-csn）为“OSS-FAULT-20250405-001”。`
	|受影响对象|可选  |用于指定受故障影响的网络资源对象。<br>例如：<br>`## 任务上下文`<br>`{{受影响对象}}`<br>其实例化示例如下：<br>`## 任务上下文 `<br>`受影响对象的ID为“BTS-001”，类型为“基站收发信台”，名称为“基站001”，位置为“北京市朝阳区”。` |
	|相关信息|可选| 其通用本体可以是与故障相关的事件列表或告警列表。<br> 例如：<br>`## 任务上下文 `<br>`{{相关信息}} `<br>其实例化示例如下：<br>`## 任务上下文`<br>`关联告警列表如下：- 告警标识符（alarm-csn）为 "ALM-20250405-001"，告警ID（alarm-id）为 "ALM-001"，告警名称为 "基站信号丢失"，网元名称为 "BTS-001"，告警位置为 "北京市朝阳区"，告警发生时间（alarm-create-time）为 "2025-04-05T14:28:00Z"。- 告警标识符（alarm-csn）为 "ALM-20250405-002"，告警ID（alarm-id）为 "ALM-002"，告警名称为 "传输链路中断"，网元名称为 "TRX-002"，告警位置为 "北京市海淀区"，告警发生时间（alarm-create-time）为 "2025-04-05T14:29:00Z"。`|
	|故障发生时间| 必选 | 其通用本体论是故障发生的时间。<br>例如：<br>`## 任务上下文 `<br>`{{相关信息}}`<br>其实例化示例如下：<br>`## 任务上下文 `<br>`故障发生时间为“2025-04-05T14:30:00Z”。` |
	| 故障上下文对象 | 必选  | 其通用本体可以是来自OSS的故障预处理信息，也可以是来自EMS的告警上报信息。<br>例如：<br>`## 任务上下文`<br>`{{故障上下文对象}}`<br>其实例化示例如下：<br>`## 任务上下文 `<br>`故障上下文对象为：“告警管理系统：FMC，告警位置：北京，告警名称：基站信号丢失，告警时间：2025-04-05T14:28:00Z，告警网元：BTS-001”。` |
    
#### 1.8.2.4 格式和规范
A2A-T中常用文本格式的语法格式规范要求，包括段落、列表、链接，使用Markdown格式确保输出的结构化和可读性：
- 段落：用空行分隔文本块
- 有序列表：数字加句点（1. 项目一）
- 无序列表：破折号开头（- 项目一）
- 链接：方括号加圆括号（[文本](链接)）

##### 1.8.2.4.1 段落
要创建一个段落，可以使用空行将一行或多行文本分隔开。样例：
```
## 任务描述
处理A社区的5G服务故障

完成服务恢复

识别故障的根本原因并进行修复
```

##### 1.8.2.4.2 有序列表
要创建有序列表，需添加以数字加句点表示的条目。这些数字不必按顺序排列，但列表应从数字1开始。样例：
```
## 预期输出 
1. Bar
2. Foo
```

##### 1.8.2.4.3 无序列表
要创建无序列表，请在每行项目前添加破折号（-）。缩进一个或多个项目以创建嵌套列表。样例：
```
## 预期输出
- 项目
  - 项目1
- Bar2
- Foo
```
##### 1.8.2.4.4 链接
要创建一个链接，请在方括号内输入链接文本，然后在其后立即加上用圆括号括起来的网址。样例：
```
## 任务描述
处理[TM Forum AN]（自治网络项目主页）无法加载的问题。
```


#### 1.8.2.5 模板定义的步骤
建议按照如下步骤定义prompt模板：
1.	确定任务类型：明确当前业务场景的任务类型和协作模式，从A2A-T任务分类体系中定位对应的模板类别（如 Task-T）；
2.	编写必要指令：选取关键指令，使用结构化的格式声明任务目标、执行条件、输入参数、预期输出等信息；
3.	填充常用变量：声明本次任务实例所涉及的具体参数，变量引用需要遵循A2A-T变量语法规范；
4.	绑定上下文信息：补充智能体完成任务所需的上下文信息；
5.	设置输出定义：明确定义输出格式、验收标准和异常处理规则；
6.	验证模板完整性：在实际跨模型环境中进行充分测试，验证语法规范性和跨LLM兼容性；
7.	版本迭代优化：将经过验证的模板纳入版本管理体系，便于持续的治理、迭代和演进。
